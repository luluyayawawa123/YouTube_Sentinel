# 《油管哨兵 / YouTube Sentinel》产品需求文档（PRD）

**副标题**：面向 Windows 的油管博主内容更新监控与 AI 简报工具  
**产品名**：油管哨兵  
**英文名**：YouTube Sentinel  
**文档版本**：v0.1（基于当前构想的整理与增强）

## 0. 文档摘要

### 产品定位
监控指定 YouTube 频道更新，自动提取内容、调用 AI 分析，并通过 Telegram 推送简报。

### 目标形态
Windows 小巧竖版客户端 + 独立后台执行器；前后台分离；纯绿色 onedir 部署。

### 本文输出
需求提取、优化建议、正式 PRD、当前版本范围和验收标准。

### 一句话定义
用户把若干频道链接放进列表后，软件在后台持续巡检；发现新作品时，自动完成内容提取、AI 总结、Telegram 推送，并在本地保留 30 天历史。

---

## 1. 需求提取与重组

### 1.1 原始需求整理后的核心模块

- **频道管理**：用户手动添加具体频道链接；软件识别、校验、保存，并展示频道列表。
- **持续监控**：后台通过 cron 定时轮询；检测到频道发布新作品后触发后续链路。
- **内容提取**：抓取标题、发布时间、描述、封面、视频链接，并尽量提取字幕、章节、简介等内容。
- **AI 分析**：对新作品生成摘要、要点、主题判断、潜在价值信息，并形成适合 Telegram 的简报。
- **消息推送**：推送到用户指定的 Telegram 会话或频道，并记录发送结果与失败原因。
- **设置中心**：所有配置项均采用“修改后点击保存”机制，并明确告知生效方式（立即 / 下次轮询 / 重启后台 / 重启前台）。
- **消息历史**：默认保存 30 天；支持分页或懒加载；支持清空历史。
- **网络诊断**：检测境外 IP、Google 首页、YouTube 首页可达状态，并显示检测时间与原始识别结果。
- **程序结构**：前后台分离；后台是唯一执行者；前台仅负责展示、配置与人工触发操作。
- **Windows 部署**：onedir 纯绿色版；通过 bat 注册 / 卸载计划任务，实现开机自启与无人登录运行。

### 1.2 建议补充为明确需求的地方

- **明确监控范围**：首版建议只监控“正式视频（Videos）”，把 Shorts、直播、首映、社区动态放到后续版本。
- **明确新作品判定**：以视频唯一 ID 为准，而不是标题或发布时间；避免改标题、补发、置顶造成误报。
- **明确内容提取优先级**：优先使用公开视频字幕或页面可得文本；若无可用文本，首版可退化为基于标题 + 描述 + 章节做轻摘要。
- **明确通知策略**：建议支持“即时推送”与“定时合并推送”两种模式，但首版默认即时推送。
- **明确失败策略**：抓取失败、AI 失败、Telegram 失败都要留痕并可重试，避免静默丢消息。
- **明确后台唯一性**：必须有单实例锁，避免计划任务与手动启动同时运行导致重复推送。

## 2. 产品判断与改进建议

### 2.1 这个方案最值得保留的设计

- **前后台分离是对的**：Windows 上后台常驻和前台展示职责不同，拆开后稳定性更高。
- **绿色 onedir 是对的**：适合中文用户的独立工具分发方式，也便于备份、迁移和故障排查。
- **保存按钮 + 生效提示是对的**：这比“边改边生效”更适合有后台任务的软件，能减少误操作。
- **保留推送历史是对的**：它既是消息记录，也是故障追踪入口。

### 2.2 建议优先优化的 10 个点

| 建议 | 原因 | 优先级 |
|---|---|---|
| 频道链接入库前统一标准化 | 保存原始链接 + 规范化频道标识，避免同一频道重复添加 | P0 |
| 首版只做视频，不做 Shorts / Live | 降低规则复杂度和误报概率 | P0 |
| 增加“立即同步一次”按钮 | 便于用户验证配置与结果 | P0 |
| 增加“测试 Telegram 推送”按钮 | 排查 Bot Token / Chat ID 最有效 | P0 |
| 后台增加单实例锁与任务去重 | 避免重复抓取、重复推送 | P0 |
| 本地数据改用 SQLite | 比纯 JSON 更适合历史、分页、状态追踪 | P0 |
| AI 分析链路做降级策略 | 没字幕时仍然能给用户一个可用简报 | P1 |
| 频道支持暂停 / 启用 | 便于临时停更或排查故障 | P1 |
| 增加运行日志 / 最近错误面板 | Windows 绿色工具必须自带可见诊断 | P1 |
| 敏感配置存储策略单独说明 | Telegram Token 等信息涉及安全与可迁移性权衡 | P1 |

### 2.3 一个重要实现建议：计划任务只负责“拉起后台”，不要负责每次轮询

更稳的方式是：Windows 计划任务只在开机时启动后台执行器；真正的监控节奏由后台进程内部的 cron 控制。这样可以统一管理状态、锁、日志、重试和热更新，而不是把每一轮巡检都交给系统任务。

### 2.4 一个重要产品建议：把“提取不到字幕”的情况产品化

YouTube 并不是每个视频都能稳定拿到高质量字幕。建议在 PRD 中明确三层提取优先级：

- **A 级**：拿到字幕 / 文本稿，做完整 AI 摘要。
- **B 级**：拿到标题、描述、章节、标签等元数据，做轻量摘要。
- **C 级**：仅拿到基础信息，则发送“新作提醒 + 基础卡片 + 待手动查看”。

## 3. 正式 PRD

### 3.1 产品概述

- **产品名称**：油管哨兵（英文名：YouTube Sentinel）。
- **产品定义**：一款面向 Windows 的桌面工具。用户添加若干 YouTube 频道链接后，后台持续巡检其更新情况；发现新作品后自动提取可用内容，调用 AI 生成摘要，并通过 Telegram 发送简报。
- **产品目标**：把“盯更新、看标题、点进去判断值不值得看、再转述给自己或别人”的流程自动化。
- **非目标（首版不做）**：账号体系、云同步、多端协同、B 站 / TikTok 等多平台支持、评论区抓取、社区帖子监控、自动下载原视频文件。

### 3.2 目标用户

- 需要持续关注多个 YouTuber 更新的重度内容消费者。
- 需要监控行业频道、资讯频道、教程频道的研究型用户。
- 希望把更新监控与 AI 摘要自动化的独立开发者或情报工作流用户。

### 3.3 核心使用流程

1. 用户首次启动软件，填写 Telegram 配置、轮询频率等设置并点击保存。
2. 用户添加一个或多个频道链接，系统校验后存入列表。
3. 后台定时巡检，发现新作品则写入待处理队列。
4. 系统提取内容、生成 AI 摘要、组装 Telegram 简报。
5. 发送成功后写入历史记录；失败则记录原因并允许重试。
6. 用户在前台查看频道状态、最近推送、网络状态与错误信息。

### 3.4 功能需求

#### 3.4.1 频道管理

- 支持添加、编辑、删除、启用、暂停频道。
- 输入必须是用户提供的具体频道链接；系统不负责模糊搜索频道。
- 保存时需进行链接校验、规范化处理，并尽量解析出频道唯一标识。
- 列表字段建议包含：频道名、原始链接、规范化标识、监控状态、最近检查时间、最近发现新作时间、最近错误状态。

#### 3.4.2 更新监控

- 后台内置 cron 计划，支持可配置的轮询周期。
- 每次巡检按频道逐个检查，产生结构化检查结果。
- 发现新作品后以视频唯一 ID 去重，保证同一视频只处理一次。
- 支持手动“立即同步一次”。

#### 3.4.3 内容提取

- 基础字段：视频标题、视频链接、发布时间、封面、描述、频道名（作者信息）。
- 增强字段：字幕、章节、标签、首条说明、作品类型（若能可靠识别）。
- 内容提取应保留原始抓取快照，便于后续重试和问题复盘。
- 提取失败时不应阻断整个流程，应走降级路径。

#### 3.4.4 AI 分析

- 输出形式至少包含：简要摘要、3–5 条核心要点、适合快速阅读的一句话结论。
- 推送中必须明确展示作者信息（至少显示频道名），避免用户只看到标题却不知道是谁更新。
- 支持固定提示词模板；首版不必开放复杂自定义 Prompt，但可以预留。
- 当文本不足时，允许生成“低置信度摘要”，并在消息中标识依据较少。

#### 3.4.5 Telegram 推送

- 支持填写 Bot Token 与 Chat ID，并提供测试发送按钮。
- 默认即时推送；发送内容格式保持简洁、适合手机阅读。
- 消息建议包含：作者/频道信息、标题、发布时间、摘要、要点、原视频链接、抓取置信提示。
- 发送失败要入历史并显示错误原因。

#### 3.4.6 设置中心

- 所有可编辑配置项统一采用“修改后点击保存”才生效。
- 保存完成后必须提示该项配置的生效机制：立即生效 / 下次轮询生效 / 需要重启后台 / 需要重启前台。
- 设置建议分组：常规、监控、AI、Telegram、历史、网络诊断、启动与后台。
- 建议增加“恢复默认值”“导出配置”“导入配置”。

#### 3.4.7 消息推送历史

- 默认保留 30 天历史记录，并定期自动清理。
- 列表支持分页或懒加载。
- 记录字段建议包含：发送时间、频道、视频标题、状态、消息摘要、失败原因。
- 提供“清除历史记录”按钮，并二次确认。

#### 3.4.8 外网状态检测

- 可检测境外网络 IP 与 Google / YouTube 首页可达状态。
- IP 检测来源包含：`https://www.whatismyip.com.tw` 与 `https://www.dnsleaktest.com`。
- 要求忠实显示识别到的原始 IP 字符串与检测时间，不擅自美化或改写。
- Google 与 YouTube 仅需简要显示可达 / 不可达 / 超时等状态。

### 3.5 非功能需求

| 维度 | 要求 |
|---|---|
| 稳定性 | 后台需具备异常恢复、单实例锁、任务去重、失败重试能力。 |
| 性能 | 前台小窗口常驻时不应明显卡顿；历史列表需支持分页或懒加载。 |
| 可维护性 | 前后台边界清晰，日志可读，配置与数据目录结构清楚。 |
| 可移植性 | 全部程序文件、配置文件、数据库文件、日志文件均放在单一目录。 |
| 可解释性 | 任何配置保存后都要明确说明生效方式。 |
| 兼容性 | 以 Windows 为首发平台，支持开机自启和无人登录后台运行。 |
| 安全性 | 敏感信息存储策略需明示；至少要防止误泄露到日志和导出包。 |

### 3.6 推荐技术架构

建议采用“Node.js 后台执行器 + Node.js / Electron 前台壳层”的双进程模式。若优先考虑开发效率与现代 UI，Electron 是更直接的选择；代价是体积较大。若未来极度追求体积，再评估其它前端壳层。

- **后台执行器**：负责 cron、抓取、去重、AI 调用、Telegram 发送、日志、网络诊断、数据清理。
- **前台界面**：负责展示状态、编辑设置、管理频道、查看历史、发起手动操作。
- **进程通信**：建议本地 HTTP API 或命名管道；前台不直接操作底层任务状态文件。
- **本地存储**：建议 SQLite + 配置文件组合；SQLite 存频道、历史、任务状态，配置文件存通用设置。

### 3.7 数据结构建议（逻辑层）

| 对象 | 关键字段 | 说明 |
|---|---|---|
| `channels` | `id, original_url, canonical_id, name, enabled, last_checked_at` | 频道主表 |
| `videos` | `video_id, channel_id, title, published_at, raw_payload, processed_status` | 新作及抓取快照 |
| `deliveries` | `video_id, tg_status, sent_at, retry_count, error_message` | 推送结果 |
| `settings` | `group_key, item_key, item_value, apply_scope` | 配置项及生效机制 |
| `history` | `type, ref_id, summary, created_at, expire_at` | 统一历史或审计记录 |

### 3.8 界面信息架构

- **主界面（竖版小窗）**：顶部状态区 + 频道列表 + 最近推送区。
- **频道管理页**：添加、编辑、暂停、删除、立即检查更新。
- **历史页**：筛选、分页 / 懒加载、清空历史。
- **设置页**：多分组表单、保存按钮、测试按钮、生效说明。
- **诊断页**：外网状态、最近错误、日志摘要、后台运行状态。

### 3.9 配置项生效规则（建议）

| 配置项类型 | 生效方式 | 提示文案建议 |
|---|---|---|
| 轮询频率 | 下次轮询生效 | 已保存，将在下一轮监控周期开始时生效。 |
| 频道列表 | 立即写库，后台刷新后生效 | 已保存，后台将在数秒内刷新频道列表。 |
| Telegram Bot Token / Chat ID | 立即校验，后续推送生效 | 已保存，建议立即点击“测试推送”验证。 |
| AI 模型 / Prompt 模板 | 下个待处理任务生效 | 已保存，将用于后续新内容分析。 |
| 界面主题 / 窗口行为 | 前台立即生效或重启前台生效 | 已保存；部分界面项可能需要重开前台。 |
| 后台启动项 | 需重启后台或重新注册计划任务 | 已保存，请按提示执行重启或重新注册。 |

### 3.10 Windows 打包与部署要求

- 打包形态：`onedir`，纯绿色，所有文件在同一目录内可见。
- 目录建议：`/app`、`/data`、`/logs`、`/config`、`/scripts`。
- 提供 bat：`register-task.bat` 与 `unregister-task.bat`。
- 计划任务应支持：开机启动、无人登录运行、失败后自动重试。
- 后台应可在无前台窗口情况下独立运行。

### 3.11 风险与对策

| 风险 | 影响 | 对策 |
|---|---|---|
| YouTube 页面结构变化 | 抓取失效或误报 | 优先基于稳定标识与多路径抓取，保留降级方案 |
| 字幕不可得 | AI 摘要质量下降 | 设计 A / B / C 三级提取与摘要策略 |
| 后台重复运行 | 重复推送、数据污染 | 单实例锁 + 任务幂等 |
| Telegram 配置错误 | 用户误以为功能失效 | 提供测试发送、错误提示与诊断页 |
| 绿色版迁移带来配置风险 | 敏感信息与兼容问题 | 单独说明配置迁移与敏感项策略 |

### 3.12 当前版本范围说明

这里原先写的 **MVP**，意思就是“**首版先做的最小可用版本**”。

更直白地说，就是：

- 先把最核心的闭环做通；
- 先不展开你当前没有明确需要的扩展项；
- 先保证这版可以稳定使用。

当前版本只聚焦以下内容：

- 频道管理
- 后台轮询
- 视频去重
- 基于标题 + 描述的基础提取
- AI 摘要
- Telegram 推送
- 30 天历史记录
- 外网状态检测

暂不在当前 PRD 中展开规划的内容：

- Shorts / Live 支持
- 自定义 Prompt
- 多平台扩展
- 合并推送
- 导入导出
- 复杂错误面板
- 手动重试

这些不是否定，只是当前版本**先不写成既定需求**，避免文档显得需求发散。

### 3.13 验收标准（首版）

- 用户可成功添加具体频道链接并保存到列表中。
- 后台在设定周期内稳定巡检，能识别同一频道的新视频且不重复推送。
- 发现新视频后，系统可生成至少一条可读 Telegram 简报并成功发送。
- Telegram 简报中必须明确包含作者信息（至少频道名），避免用户无法判断是谁更新。
- 设置页所有修改都需点击保存才生效，并给出明确生效提示。
- 历史记录默认保留 30 天，支持分页 / 懒加载与手动清空。
- 软件可在 Windows 上以 onedir 绿色版运行；计划任务可拉起后台，且在无用户登录时继续运行。
- 外网状态检测页能展示两个站点识别到的 IP 与 Google / YouTube 可达结果。

### 3.14 当前仍需拍板的开放决策

- 首版是否只监控 Videos，不监控 Shorts / Live。
- AI 摘要允许的成本区间和模型档位。
- Telegram 是只支持单一目标 Chat，还是允许多目标。
- 敏感配置是优先“可迁移”还是优先“本机加密”。
- 前台技术壳层是否接受 Electron 带来的体积。

---

## 4. 我的结论

你的原始构想已经足够完整，真正要补的是“边界”和“降级”。只要你把首版范围收紧为：**指定频道 + 新视频监控 + 文本可得时 AI 摘要 + Telegram 推送 + Windows 绿色常驻**，那么这就是一个很有落地性的工具。
